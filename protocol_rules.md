# FORGE Protocol Generation Rules

This document supports the existing FORGE Intelligence layer in `C:\FORGE` and the repository-level [AGENTS.md](C:/FORGE/Internal%20Platform/FORGE%20CSS/AGENTS.md).

It is not a replacement for the governing FORGE system.

## Core Standard

- Continuity over reinvention.
- Upgrade from what already exists before inventing something new.
- Prefer summaries and extracted key data before raw documents.
- Read full uploaded documents only when summaries are insufficient or explicit review is required.
- Use prior protocol structure, progression intent, and continuity signals as the default starting point.

## Required Document Review Order

Review source context in this order before writing:

1. Most recent protocol summary
2. Original or baseline protocol summary
3. Journal, check-in, and adherence summary
4. Coach notes summary

If summaries are incomplete, selectively inspect the underlying document that matters most.
If uploaded documents conflict, prioritize the most recent protocol plus the most recent behavior and recovery evidence.

## Extract Before Writing

Before generating the next protocol, extract:

- prior phase
- prior split
- prior movement priorities
- prior cardio
- prior abs/core
- progression intent
- current recovery constraints
- current adherence constraints

## Document Continuity Check

Every generated protocol must begin with a **Document Continuity Check** that states:

- what source documents or summaries were reviewed
- what the prior protocol was trying to accomplish
- what continuity elements must be preserved
- what current recovery or adherence constraints are active

If progression data are incomplete, state exactly:

`Insufficient progression data in uploaded documents - using conservative continuation logic.`

## Continuity Decisions

Before finalizing a protocol, determine:

- what to continue
- what to progress
- what to reduce
- what to replace
- what to hold constant

These decisions must be explicit, not implied.

## Protocol Continuity Rules

- Do not reset to a generic template if a prior structure exists.
- Do not remove movement specificity unless documents justify it.
- Do not change split structure, cardio structure, or core structure without a documented reason.
- When adherence or recovery drops, reduce chaos, not specificity.
- When data are incomplete, prefer conservative continuation over aggressive change.
- Preserve internally coherent macro-to-meal translation.

## Physique Competitor Rules

For physique competitors, especially NPC bikini athletes:

- Preserve bodybuilding specificity.
- Reduce chaos, not specificity.
- Progress from prior structure instead of resetting to a generic template.
- Maintain realistic starting weight ranges based on the prior protocol, current body composition, and recovery context.

### Movement

- Keep glute and hamstring priority central.
- Maintain delt and upper-back emphasis to support waist illusion.
- Preserve physique-driven exercise intent even when simplifying execution.
- Use progression from prior split, prior movement priorities, and prior progression intent.

### Waist Presentation

- Avoid excessive oblique hypertrophy.
- Abs and core work should support control, posture, bracing quality, and tighter waist presentation without thickening the waist.

### Cardio

- Cardio should support body composition without interfering with lower-body recovery.
- Cardio changes should be justified against current recovery, leg training demands, and adherence.

## Progression Standard

When progressing from a prior protocol:

- advance only what the documents support
- preserve effective anchors
- avoid novelty for its own sake
- justify any regression as intentional
- justify any lateral hold as a continuity decision, not an omission

## Output Standard

A protocol is not ready unless it shows:

- document continuity
- clear carry-forward logic
- progression or justified regression
- sport-specific coherence
- realistic execution
